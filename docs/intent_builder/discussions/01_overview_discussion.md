## 已明确的设计决策

### Q1: 意图的粒度应该如何定义？

**结论**: 粗粒度，基于页面状态变化切分

**说明**:
- URL 变化 → 新意图
- 在同一页面完成一个语义完整的子任务 → 一个意图
- 避免过细（每个点击一个意图）和过粗（整个任务一个意图）

---

### Q2: 如何推断循环？

**结论**: 从用户自然语言描述中推断，关键词检测

**说明**:
- MVP 阶段通过检测关键词（"所有"、"每个"、"遍历"、"all"、"every"等）
- 不从意图连线中自动推断循环（未来版本考虑）

---

### Q3: 意图是否需要多版本管理？

**结论**: MVP 不考虑多版本，只保留第一个版本

**说明**:
- 简化实现
- 未来版本可以支持版本优化和选择

---

### Q4: 用户是否需要交互式修改 MetaFlow？

**结论**: MVP 不支持，未来版本支持

**说明**:
- MVP 完全自动生成
- 未来支持：用户查看 MetaFlow → 通过自然语言描述 → LLM 修改 MetaFlow
- 当前阶段暂不考虑这个功能

---

### Q5: 意图记忆图的存储方案？

**结论**: MVP 使用内存存储

**说明**:
- Python 字典实现
- 或使用 Python 图库（如 NetworkX）
- 持久化在后期引入（SQLite 图引擎或其他方案）

---

### Q6: MVP 范围界定

**包含的功能**:
- 意图提取（粗粒度）
- 意图记忆图构建（内存）
- 意图检索（标签匹配 + 频率排序）
- 线性 MetaFlow 生成（+ 简单循环）
- YAML 生成和执行

**不包含的功能**:
- 意图多版本管理
- 交互式 MetaFlow 修改
- 从意图连线自动推断循环
- 复杂控制流（条件分支、嵌套循环）
- 向量检索（只用标签匹配）
- 持久化存储
