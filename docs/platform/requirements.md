## 1. 产品概述

### 1.1 产品定位

**Ami** 是首个通过观察学习的可进化 AI Agent。用户只需正常工作一次，Ami 就能学会如何自动化重复性任务，并持续优化和进化。

**核心理念**：
> 不是让用户学会如何使用工具，而是让 AI 学会用户如何工作。

### 1.2 问题陈述

知识工作者每周花费 **12-15 小时**在重复性数字任务上：
- 跨多个网站采集数据
- 填写重复性表单
- 导出和整理报告

**现有解决方案的问题**：
- **通用 AI Agent**：不可靠（成功率<70%），昂贵（$1-2/任务），需要精心设计提示词
- **传统 RPA**：脆弱（页面改版就失效），零复用性，开发和维护成本高

**根本原因**：
忽略了最可靠的智能来源 —— **人类专家的真实工作经验**

### 1.3 解决方案

**Ami 的核心能力**：
1. **观察学习**：用户正常工作一次，Ami 自动学习
2. **语义理解**：理解操作背后的意图，而非死记硬背
3. **智能复用**：将工作流分解为可复用的 Intent 单元
4. **持续进化**：使用越多，系统越智能

**关键指标**：
- ✅ 成功率：>95%（已学会的场景）
- 💰 成本：$0.02-0.05/任务（降低95%）
- 🚀 学习曲线：零（用户只需正常工作）

## 4. 功能需求

### 4.1 用户注册与认证

**FR-1.1 用户注册**
- 用户可以通过邮箱注册账号
- 需要验证邮箱有效性
- 密码强度检查（至少 8 位，包含字母和数字）

**FR-1.2 用户登录**
- 支持邮箱 + 密码登录
- 登录后获得 JWT token（30 分钟有效期）
- Desktop App 和 Extension 共享登录状态

**FR-1.3 会话管理**
- Token 自动刷新
- 30 分钟无操作自动登出
- 多设备登录支持

---

### 4.2 操作录制

**FR-2.1 开始录制**

**触发方式**：
- Extension popup 点击"开始录制"按钮

**前置条件**：
- Desktop App 已启动（Backend 运行在 localhost:8000）
- 用户已登录
- 没有其他正在进行的录制

**系统行为**：
1. Extension 发送请求到本地 Backend
2. Backend 创建新的 recording session
3. 返回 session_id 给 Extension
4. Extension 显示"录制中"状态（红色指示灯）
5. 开始捕获用户操作事件

**后置条件**：
- 录制会话已创建
- Extension 处于录制模式

**FR-2.2 捕获操作**

**支持的操作类型**：
- ✅ 点击（click）
  - 捕获：元素选择器、坐标、时间戳
- ✅ 输入（input）
  - 捕获：输入框选择器、输入值、时间戳
- ✅ 导航（navigate）
  - 捕获：目标 URL、来源 URL、时间戳
- ✅ 滚动（scroll）
  - 捕获：滚动位置、方向、时间戳
- ✅ 选择（select）
  - 捕获：下拉框选择器、选中值、时间戳

**暂不支持（MVP）**：
- ❌ 鼠标悬停（hover）
- ❌ 拖拽（drag & drop）
- ❌ 键盘快捷键
- ❌ 右键菜单

**数据存储**：
- 实时发送到本地 Backend
- Backend 追加到 `operations.json`
- 本地文件路径：`~/ami/storage/users/{user_id}/learning/{session_id}/operations.json`

**FR-2.3 录制状态显示**

Extension 实时显示：
- 录制时长（秒）
- 已捕获操作数量
- "停止录制"按钮（醒目、易点击）

**FR-2.4 停止录制**

**触发方式**：
- Extension popup 点击"停止录制"按钮

**系统行为**：
1. Extension 发送"停止"请求到 Backend
2. Backend 标记 recording session 为"已完成"
3. Backend 保存最终的 operations.json
4. Extension 显示"录制已完成，正在上传..."

**后置条件**：
- 录制会话已结束
- 数据已保存到本地
- 自动进入"上传"阶段

---

### 4.3 数据上传与处理

**FR-3.1 上传录制数据**

**触发时机**：
- 用户点击"停止录制"后自动触发

**系统行为**：
1. Backend 读取 operations.json
2. 调用云端 API：`POST /api/recordings/upload`
3. 显示上传进度（可选）
4. 上传成功后，云端返回 recording_id

**容错**：
- 上传失败 → 自动重试（最多 3 次）
- 仍失败 → 提示用户"上传失败，请检查网络连接"
- 本地数据保留，用户稍后可重新上传

**FR-3.2 云端生成 Workflow**

**触发时机**：
- 上传成功后自动触发

**系统行为**（云端）：
1. 调用 LearningService.extract_intents(operations)
2. 调用 LearningService.generate_metaflow(intents)
3. 更新用户的 Intent Graph
4. 调用 WorkflowService.generate_workflow(metaflow)
5. 保存 workflow.yaml 到云端
6. 返回 workflow_name

**预计耗时**：30-60 秒

**用户体验**：
- 显示模态窗口（同步等待）
- 进度指示：
  ```
  正在生成 Workflow...
  ✅ 分析操作意图 (完成)
  ⏳ 构建 MetaFlow (进行中 60%)
  ⏸️  生成 Workflow (等待)
  
  预计剩余时间：20 秒
  ```

**容错**：
- LLM API 调用失败 → 显示"生成失败，请稍后重试"
- YAML 验证失败 → 显示"Workflow 生成异常，请联系支持"

**FR-3.3 下载 Workflow 到本地**

**触发时机**：
- 云端生成完成后自动触发

**系统行为**：
1. Backend 调用云端 API：`GET /api/workflows/{name}/download`
2. 下载 workflow.yaml
3. 保存到本地：`~/ami/storage/users/{user_id}/workflows/{workflow_name}/workflow.yaml`
4. 保存 metadata.json（包含创建时间、描述等）

**后置条件**：
- Workflow 已保存到本地
- 用户可以执行

**FR-3.4 自动命名**

**命名规则**：
- 云端 LLM 根据操作内容生成描述性名称
- 示例：
  - "从 Allegro 抓取咖啡产品数据"
  - "登录 Amazon 并搜索电子产品"
  - "导出销售数据到 Excel"

**用户可修改**：
- Extension 或 Desktop App 显示生成的名称
- 用户可以点击编辑，自定义名称
- 名称唯一性检查（同名加后缀 _1, _2）

### 4.4 Workflow 执行

**FR-4.1 查看 Workflow 列表**

**触发方式**：
- Extension popup 显示"我的 Workflows"
- Desktop App 主界面显示 Workflow 列表

**列表内容**：
- Workflow 名称
- 创建时间
- 最后执行时间
- 执行次数
- 成功率

**操作**：
- 点击名称 → 查看详情
- 点击"执行"按钮 → 立即执行
- 点击"删除"按钮 → 删除 Workflow

**FR-4.2 执行 Workflow**

**触发方式**：
- Extension 或 Desktop App 点击"执行"按钮

**前置条件**：
- Workflow 文件存在于本地
- Backend 正在运行
- 全局浏览器会话已初始化

**系统行为**：
1. Extension/App 调用本地 Backend API：`POST /api/workflows/{name}/execute`
2. Backend 加载 workflow.yaml
3. 设置 `workflow.name = "global"`（复用全局浏览器）
4. BaseAgent 执行 workflow
5. 返回 task_id

**执行过程**：
- Extension 轮询 task 状态（每 2 秒）
- 显示执行进度（当前步骤、进度百分比）
- 实时更新状态："执行中..." → "完成" / "失败"

**后置条件**：
- 执行结果已保存
- 用户可以查看结果

**FR-4.3 显示执行结果**

**成功情况**：
```
┌────────────────────────────────┐
│  ✅ 执行成功！                  │
├────────────────────────────────┤
│  Workflow: 从 Allegro 抓取咖啡  │
│  执行时间: 2 分钟 15 秒         │
│  结果: 已抓取 50 条数据         │
│                                │
│  [📥 下载结果] [🔁 重新执行]    │
└────────────────────────────────┘
```

**失败情况**：
```
┌────────────────────────────────┐
│  ❌ 执行失败                    │
├────────────────────────────────┤
│  错误: 页面结构已变化           │
│  失败步骤: 第 3 步（点击搜索）  │
│                                │
│  建议: 请重新录制该 Workflow    │
│                                │
│  [🔴 重新录制] [📋 查看详情]    │
└────────────────────────────────┘
```

**FR-4.4 管理 Workflow**

**查看详情**：
- Workflow 名称、描述
- 创建时间
- 包含的步骤（简要列表）
- 执行统计（总次数、成功次数、失败次数）

**删除 Workflow**：
- 用户确认 → 删除本地文件
- 云端备份保留（用于数据分析）

## 5. 非功能需求

### 5.1 性能要求

**NFR-1 响应时间**
- Extension UI 操作响应：< 100ms
- 录制操作捕获延迟：< 50ms
- Workflow 执行启动时间：< 2s
- 云端 Workflow 生成：30-60s（可接受）

**NFR-2 资源占用**
- Desktop App 内存占用：< 500MB
- Backend 内存占用：< 200MB
- CPU 占用（空闲时）：< 5%

**NFR-3 并发支持**
- 单用户同时执行 Workflow：1 个（全局浏览器限制）
- 云端同时处理生成请求：10 个（MVP）

### 5.2 可靠性要求

**NFR-4 可用性**
- 本地 Backend 正常运行时间：> 99%
- 云端 API 可用性：> 99.5%

**NFR-5 数据持久化**
- 录制数据本地保存：100%（即使上传失败）
- Workflow 本地保存：100%
- 执行结果保存：100%

**NFR-6 容错**
- 网络断开：本地数据不丢失，恢复后自动重试
- Backend 崩溃：自动重启（Desktop App 监控）
- LLM API 失败：友好错误提示，支持重试

### 5.3 安全性要求

**NFR-7 数据传输**
- 所有 API 通信使用 HTTPS
- JWT Token 过期时间：30 分钟
- 敏感数据（密码）使用 bcrypt 加密

**NFR-8 数据存储**
- 本地文件权限：用户私有（chmod 600）
- 云端数据：用户隔离（不可跨用户访问）

**NFR-9 隐私保护**（MVP 暂不实现，记录为已知限制）
- 录制数据包含用户输入值（可能包含敏感信息）
- MVP 不做脱敏处理
- 需在用户协议中说明

### 5.4 兼容性要求

**NFR-10 平台支持**
- macOS 12+（优先）
- Windows 10+（V2）
- Linux（暂不支持）

**NFR-11 浏览器支持**
- Chrome 100+（必须）
- Edge（基于 Chromium，理论支持）
- Firefox、Safari（暂不支持）

**NFR-12 分辨率支持**
- 最小分辨率：1280x720
- 推荐分辨率：1920x1080

### 5.5 可维护性要求

**NFR-13 日志**
- Backend 记录详细日志（DEBUG 级别）
- 日志自动滚动（每日 / 10MB）
- 错误自动上报云端（可选，用户同意）

**NFR-14 版本管理**
- Desktop App 自动检查更新
- 提示用户下载新版本
- 向后兼容（旧版 workflow 仍可执行）

## 6. 用户故事

### US-1 市场分析师抓取价格数据

**作为** 市场分析师李明  
**我想要** 自动化从 Allegro 抓取咖啡产品价格的流程  
**以便于** 每周节省 5 小时手动数据采集时间

**场景**：
```
Given 我已安装 Ami Desktop App 和 Extension
And 我已登录账号
When 我打开 Allegro.pl
And 点击 Extension "开始录制"
And 我执行以下操作：
  - 在搜索框输入 "kawa"（咖啡）
  - 点击"搜索"
  - 选择筛选条件（价格、评分）
  - 点击"下一页"翻页
  - 复制产品信息
And 点击 Extension "停止录制"
Then 系统自动上传数据到云端
And 30-60 秒后生成 Workflow "从 Allegro 抓取咖啡产品数据"
And 我点击"执行"按钮
Then Workflow 自动执行，抓取数据
And 我可以下载结果 Excel 文件
```

**验收标准**：
- ✅ 录制完整无遗漏
- ✅ 生成的 Workflow 名称准确描述操作
- ✅ 执行成功率 > 95%
- ✅ 执行时间 < 5 分钟

---

### US-2 复用 Workflow 搜索不同产品

**作为** 市场分析师李明  
**我想要** 用已有的 Workflow 搜索茶叶产品（而不是咖啡）  
**以便于** 无需重新录制，直接复用

**场景**：
```
Given 我已有 Workflow "从 Allegro 抓取咖啡产品数据"
When 我在 Extension 点击该 Workflow
And 选择"修改参数"（V2 功能，MVP 暂无）
Then 系统提示："MVP 暂不支持参数化，请重新录制"
```

**MVP 限制**：
- ❌ 不支持参数化
- 🔄 V2 计划支持

---

### US-3 处理执行失败

**作为** 用户  
**我想要** 当 Workflow 执行失败时，知道为什么以及如何解决  
**以便于** 快速恢复自动化流程

**场景**：
```
Given 我有 Workflow "从 Allegro 抓取咖啡产品数据"
And Allegro 网站改版，页面结构变化
When 我执行该 Workflow
Then 系统显示错误："页面结构已变化，无法找到搜索按钮"
And 提示："建议重新录制该 Workflow"
And 提供"重新录制"按钮
When 我点击"重新录制"
Then 自动开始新的录制流程
```

**验收标准**：
- ✅ 错误信息清晰、可理解
- ✅ 提供明确的解决建议
- ✅ 一键重新录制

---

### US-5 多设备同步

**作为** 用户


## 附录

### A. 术语表

| 术语 | 定义 |
|------|------|
| **Operation** | 用户的一次操作（如点击、输入） |
| **Recording** | 一次录制会话，包含多个 Operations |
| **Intent** | 语义化的操作单元，可复用 |
| **Intent Graph** | 用户的 Intent 知识图谱 |
| **MetaFlow** | Intent 到 Workflow 的中间表示 |
| **Workflow** | 可执行的自动化流程（YAML 格式） |
| **BaseAgent** | 执行引擎 |

### B. 已知限制（MVP）

1. **不支持敏感数据处理**：录制可能包含密码等，暂无脱敏
2. **不支持 iframe 跨域**：某些页面无法完全捕获
3. **不支持验证码**：需要用户手动处理
4. **不支持参数化**：每个场景需单独录制
5. **单浏览器限制**：只支持 Chrome
